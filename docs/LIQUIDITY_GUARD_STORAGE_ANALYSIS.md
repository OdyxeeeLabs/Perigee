# Liquidity Pool Guard Storage Gas Optimization Analysis

**Date**: May 26, 2026
**Issue**: #218 - Profile storage gas optimizations for liquidity_pool guard integration
**Goal**: Confirm expected 87.5% storage efficiency gain when replacing multiple boolean pause flags with a single `u32` bitmask (`PauseState`).

## Summary

- Replacing 8 separate boolean pause flags with a single `u32` (`PauseState`) stores 8 pause states in 4 bytes instead of ~160 bytes (8 × ~20 bytes storage entry overhead).
- Storage reduction for pause flags: (160 - 20) / 160 = 0.875 → 87.5%.
- Per-check cost: one storage read + bitwise ops vs multiple storage reads; large gas savings on hot paths like `deposit`, `swap`, `withdraw`.

## Baseline: Boolean flags

Assume 8 pause flags (one per operation) stored as separate instance keys. Each instance entry costs ~20 bytes (data + overhead).

Total: 8 × ~20 bytes = ~160 bytes for pause state storage.

Per-check cost: up to 8 separate storage reads when checking multiple flags.

## Optimized: Bitmask `PauseState` (u32)

Single entry: `PauseState` = `u32` (4 bytes) + instance overhead ≈ ~20 bytes total.

Per-check cost: 1 storage read + cheap bitwise AND for each flag.

## Storage Savings Calculation

```
Baseline (8 flags): ~160 bytes
Bitmask (u32): ~20 bytes
Savings: 160 - 20 = 140 bytes
Percentage: 140 / 160 = 87.5%
```

## Gas Impact (Qualitative)

- Hot functions (`deposit`, `swap`, `withdraw`) each call `check_not_paused`. Replacing multiple reads with a single read reduces storage I/O significantly.
- Expected per-call gas reduction: substantial (depends on ledger pricing), cumulative savings high for high-volume pools.
